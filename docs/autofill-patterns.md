# Autofill Runtime Patterns

Known patterns that autofill scripts handle:

- **Ashby Boolean hidden checkbox:** Yes/No button clicks only add CSS `_active` class â€” the hidden `<input type="checkbox">` must be force-set via JS after button click
- **Ashby reCAPTCHA spam rejection:** "We couldn't submit your application" is detected as `captcha_required` for graceful skip
- **Ashby Number/Boolean types:** LLM answers for Number fields extract digits; Boolean fields map to Yes/No buttons
- **Greenhouse hidden checkboxes:** Acknowledgment/consent checkboxes use custom SVG styling with native input hidden â€” use `force=True` for checking
- **Greenhouse demographic labels:** Click labels (not force-check inputs) for single-select demographics to trigger Greenhouse JS handlers
- **Greenhouse GDPR consent:** Selectors use `*=` (contains) matching for `gdpr_demographic_data_consent` to handle `job_application[data_compliance][gdpr_...]` wrappers
- **Greenhouse acknowledgment fields:** `_is_acknowledgment_field` guards prevent "current employer" false positives when the only option is "I acknowledge"
- **Greenhouse city/location dropdown:** "In what cities are you available to work" and similar city-selection questions are matched deterministically via `_location_option_label`. Logic: select the role's designated city from `jd_parsed.json` location field first (candidate is always available wherever the role is) â†’ fall back to candidate's city (San Francisco) if role location unknown â†’ last resort, first non-Remote option
- **React textarea re-render clearing:** Choice/button fills after textarea fills trigger React re-renders that clear textarea values. Pipeline Phase 2b re-verifies and re-fills LongText fields after all steps complete
- **Transient validation errors:** Forms may briefly flash validation_error before navigating to success page. Pipeline waits 2s and re-checks before breaking on validation_error
- **Greenhouse deterministic questions:** "preferred first/last name" answered from `master_resume.md`; standalone "ethnicity" labels handled as demographic questions from `application_profile.md`
- **Salary comfort vs. salary expectation questions:** `question_is_salary_comfort_check` uses a two-step gate: long-form fragments (e.g. "comfortable with the salary range") match immediately; short keyword fragments (e.g. "salary range") require a secondary "comfortable" or "interview" keyword. This prevents open-ended salary expectation questions ("What is your desired salary range?") from being falsely answered "Yes". Open-ended salary/compensation questions are caught by `_question_is_compensation` and overridden with negotiable language.
- **Shared compensation and undergraduate GPA defaults:** explicit free-text salary/compensation expectation prompts now resolve through `resolve_shared_question_policy()` using the profile-backed compensation answer from `application_profile.md`, and explicit undergraduate/Bachelor's GPA prompts resolve to the profile-backed `Undergraduate GPA` value (`3.8/4.0`). If either shared value is missing, the submitter fails closed through the existing pending-user-input / planned-unconfirmed artifact path instead of dropping the field or drifting into LLM output.
- **Pronouns demographic handler:** Ashby, Greenhouse, Lever, Gem all have explicit handlers mapping `application_profile.pronouns` ("He / Him / His") to multiple candidate forms ("He/Him", "He/Him/His", etc.) for matching. Workday/Phenom/iCIMS handle via standard EEO fields or LLM path.
- **Race/ethnicity Hispanic normalization:** Ashby and Greenhouse expand "Hispanic or Latino" to multiple candidate forms ("Hispanic/Latino", "Latinx", "Hispanic or Latino/a/x", etc.) for matching MultiValueSelect checkbox groups. Other boards handle natively or via LLM.
- **Product usage questions â†’ always "Yes":** `_question_is_product_usage` in `application_submit_common.py` detects "Have you used [product]?" patterns (excludes open-ended "describe your experience" variants). Classified pre-generation by `classify_question()` in `question_classifier.py` so they never reach the LLM. Board-specific `_infer_step` handlers in Ashby, Lever, Gem also handle them at fill time.
- **Combobox already-filled detection:** Skip comboboxes with non-placeholder values to avoid overwriting valid LLM-filled answers
- **Greenhouse "I identify as:" demographic:** Mapped to `transgender_status` (not gender). Fragment placed after "gender" in `demographic_question_specs` to prevent premature matching of "I identify my gender as:" labels. Keyboard typing fallback attempts if dropdown click approach fails for demographic comboboxes.
- **Workday CXS API JD scraping:** Workday SPA pages often return empty shells to scrapling/stealth fetchers. `scrape_job.py` falls back to the Workday CXS JSON API (`/wday/cxs/{company}/{site}/job/...`) which returns structured job posting data. Works for both `myworkdayjobs.com` and `myworkdaysite.com` patterns
- **Phenom multi-page wizard:** 6 possible steps (My Information â†’ My Experience â†’ Application Questions â†’ Voluntary Disclosures â†’ Self Identity â†’ Review). Not all instances have all steps. Uses `autofill_main` with custom `run_browser_fn` callback (like Workday), not `run_browser_pipeline`
- **Phenom React textarea filling:** `fill()` and native event dispatching don't trigger React validation. Use `type()` (keyboard simulation) as primary strategy, with React fiber `onChange` direct call as backup. Truncate descriptions to 3 bullets to keep `type()` fast
- **Phenom select option matching:** Priority-based scoring (exact > starts-with > contains > reverse-contains, tie-break by shorter text length) to avoid wrong matches like "Agricultural Business" when "Business Administration" is available
- **Phenom talent community redirect:** After completing all wizard pages, some instances redirect to "Join our Talent Community" URL. Detected via URL containing `talentcommunity` and treated as successful submission. The early-exit confirmation path must call `write_report()` with `all_filled` data before returning, otherwise the autofill report will be empty.
- **Confirmation email search:** The fallback Gmail search includes `submission OR submitted OR "thank you"` keywords (not just `application OR applying OR received`) to match varied confirmation email subjects (e.g. HPE's "Thank you for your online submission"). Company name searches body text, so avoid false positives from common words (e.g., "circle back" matching a search for "Circle")
- **Conditional follow-up regex:** `_CONDITIONAL_FOLLOWUP_RE` matches "If you answered 'Yes'", "the previous question", "If sponsorship is required, please confirm", and other conditional patterns. Optional conditional follow-ups are left blank; required ones get "N/A". Deterministic overrides (e.g. sponsorship) must also check `_is_conditional_followup` before filling â€” otherwise a conditional question like "If you require visa sponsorship, indicate type" gets the full sponsorship answer instead of "N/A"
- **Greenhouse hispanic/latino dropdown:** The `#hispanic_ethnicity` selector matches the container div, not the `<select>`. Use `select#job_application_hispanic_ethnicity` or `#hispanic_ethnicity select`. Demographic group scan must include `[data-eeoc-question]` containers (not just `.demographic_question`) to find the hispanic dropdown
- **Company engagement questions:** Classified pre-generation by `classify_question()` in `question_classifier.py`, which routes "engaged with" / "interacted with" / "been exposed to" questions to deterministic answers (product/course/platform options, then blog/content fallback) before they reach the LLM. Board-specific `_infer_step` handlers also select the correct option at fill time.
- **Deterministic questions excluded before LLM generation:** `classify_question()` in `question_classifier.py` identifies deterministic questions (product usage, company engagement, salary comfort, work authorization, etc.) and routes them to pre-set answers before LLM generation, so they never consume provider tokens. Draft overrides (user edits from the draft review flow) still apply via `apply_draft_overrides()`.
- **Shared positive-fit screening policy:** `resolve_shared_question_policy()` in `application_submit_common.py` sits between `classify_question()` and board-local option picking. It answers discrete fit-screening prompts affirmatively across boards (hybrid/in-office/commute, relocation, travel, product usage, minimum-years, general experience/background) while keeping work authorization, sponsorship, compensation, and self-ID on their existing truthful paths. Degree/license/certification claims only answer `Yes` when `application_profile.md` or explicit credential lines in `master_resume.md` support them.
- **Demographic select pre-selected defaults:** Phase 2 demographic scan must not skip a select just because it has a non-placeholder value. Must check if the current value matches the desired value â€” pre-selected defaults like "No" for hispanic must be overridden to match the profile
- **Phenom former employee radio:** Clicks "No" label directly for "Have you previously been employed" patterns. Handles both `value="true"/"false"` and `value="Yes"/"No"` radio variants
- **Workday combobox interaction:** Workday uses React-based comboboxes (How Did You Hear, State) that don't respond to Playwright's `click(force=True)` or `check()`. Must find the input element inside the form field container via JS, type to trigger the dropdown, then click options. After selection, force-close popup overlays (set `display:none` on `[data-behavior-click-outside-close]` elements). Radio buttons need label clicks, not input clicks.
- **Workday popup overlay blocking:** After filling Workday comboboxes, the dropdown popup can overlay subsequent form fields. Must dismiss via Escape + JS force-close before filling other fields.
- **Company-hosted jobs pages can be TalentBrew/iCIMS or Jobs2Web/SuccessFactors:** custom `jobs.*` hosts are not one family by hostname alone. Route to iCIMS only when the HTML exposes `icims.com` or `talentbrew`; route to SuccessFactors when the page exposes Jobs2Web markers such as `jobs2web`, `j2w.apply`, or SuccessFactors application assets.
- **Recruitee company wrappers:** company-hosted `/o/...` pages should route to `recruitee` when the live HTML exposes Recruitee assets such as `cdn.recruitee.com/assets/app.js`. Marketing or landing pages without a role path should stay unsupported.
- **Breezy direct hosts:** treat `*.breezy.hr/p/...` pages as direct Breezy application surfaces and keep the board-specific submitter on the standard single-page pipeline.
- **JazzHR / ApplyToJob routing:** `applytojob.com/apply/...` pages are JazzHR-backed and should route to the `jazzhr` submitter rather than staying in the unsupported bucket.
- **Paycor Recruiting routing:** `recruitingbypaycor.com/Recruiting/Jobs/...` pages are first-class draft-mode surfaces and should route to the `paycor` submitter.
- **Wave 1 simple ATS families share allowed infrastructure layers but keep explicit board contracts:** Breezy, Recruitee, Jobvite, JazzHR, and Paycor now reuse shared payload helpers in `application_submit_common.py`, shared fill/classification helpers in `autofill_common.py`, and the shared runtime wrapper in `autofill_pipeline.py`, while each board keeps its own name, selectors, submit button set, and confirmation/review detection.
- **Aggregator URL resolution:** `url_resolver.py` resolves aggregator URLs (LinkedIn, Indeed, Glassdoor) to board URLs before scraping. Both `run_pipeline.py` and `pipeline_orchestrator.py` call `resolve_to_board_url()`. Uses HTTP HEAD redirects first, then browser-based resolution with persistent LinkedIn profile at `.playwright-linkedin/`. Set up session via `job-assets linkedin-login`.
- **LinkedIn mark-as-applied:** After successful submission of a LinkedIn-sourced job, `_post_submit()` in `pipeline_orchestrator.py` calls `mark_linkedin_job_applied()` from `url_resolver.py`. This navigates to the LinkedIn job page using the persistent profile, clicks the Apply button to trigger the "Did you apply?" dialog, and clicks Yes. Best-effort â€” failures are logged but don't change job status. Uses the same file lock as URL resolution to prevent profile contention.
- **Full-page screenshot sticky header fix:** `capture_screenshot()` in `autofill_common.py` temporarily converts `position: fixed/sticky` elements to `position: absolute` before taking `full_page=True` screenshots. This prevents sticky headers (common on Greenhouse, Ashby) from repeating at every scroll position in the full-page capture.
- **Rippling resume auto-parse:** Rippling auto-fills some fields (name, email) after resume upload. The `_fill_step` handler fills fields regardless, overwriting any parsed values with canonical profile data. EEO fields (gender, race, veteran, disability) are optional and skipped.
- **Rippling listingâ†’apply navigation:** Job listing URLs (`ats.rippling.com/{company}/jobs/{id}`) need `/apply` appended. `_navigate_to_apply_form` clicks "Apply now" button if not already on the apply page.
- **Uber account creation/sign-in:** `autofill_uber.py` handles account lifecycle: tries "Create account" first (email+password), falls back to "Sign in" if account exists. Credentials via `UBER_EMAIL` / `UBER_PASSWORD` env vars. Form structure discovered at runtime after auth; unknown fields handled via LLM answering.
- **Motion Recruitment modal form:** Application form opens in a modal after clicking "Apply" on the job listing page. `_wait_for_motion_form` dismisses cookie banner, opens the modal, then waits for form render. reCAPTCHA is present â€” handled via standard captcha wait in headed mode.
- **Reducto inline form:** Application form is embedded at the bottom of the job listing page (no separate URL). `_wait_for_reducto_form` scrolls to the form. Radio questions (onsite, startup, sponsorship) are deterministically answered.

## Implementation Rules

These are standing implementation rules for all autofill/submit work. They govern how boards are built, how submissions are confirmed, and how edge cases are handled.

### General Workflow
- For every durable code change: (1) generalize across all supported boards (Greenhouse, Gem, Lever, Ashby, Dover, Workday, Phenom, iCIMS, SuccessFactors, Breezy, Recruitee, Jobvite, JazzHR, Paycor, and the rest of the active board set) unless provably board-specific, (2) update `AGENTS.md`, regenerate all provider copies with `uv run python scripts/sync_agent_files.py`, and refresh any affected workflow docs such as `README.md`, (3) commit, push, merge to `main`.
- Fix live bugs during application runs immediately; generalize and update tests in the same change.
- Keep repo clean: version durable files, `.gitignore` transient/debug output.
- `codex/*` branch pushes open/reuse a PR targeting `main` with auto-merge.
- Treat multi-page application flows as first-class: capture page-by-page screenshots and surface unresolved questions in artifacts when automation cannot answer them confidently.

### Greenhouse
- For Greenhouse submit attempts, handle both modern and classic embed flows, including same-page confirmation states. If submit does not reach a confirmed completion state, write debug HTML and screenshot artifacts before failing.
- For Greenhouse proof-gap exits after payload generation, write `submit/application_submission_result.json` with a Greenhouse-specific `failure_type` such as `greenhouse_unknown_questions`, `greenhouse_review_proof_gap`, `greenhouse_submit_validation`, `greenhouse_submit_not_confirmed`, or `greenhouse_submit_navigation_missing`, and include the current submit-debug / unknown-question artifacts that actually exist on disk.
- For Greenhouse email-verification flows that appear only after the first submit click, treat the visible security-code field as the next expected step instead of a generic validation failure. After entering the code, re-check the live page for late confirmation navigation before declaring the submit attempt failed.
- For Greenhouse verification-code fetches, scope the Gmail lookup to the current submit attempt and wait briefly for the new message so stale codes from earlier retries are not reused.
- For Greenhouse custom career pages that embed the application via `gh_jid` query params (e.g. `careers.toasttab.com/jobs/...?gh_jid=123`), discover the correct Greenhouse board slug by probing the API with candidate slugs derived from the domain (skipping generic subdomains like "careers", "jobs", "apply") and the pipeline's company slug. Do not blindly use the hostname prefix as the board slug.
- For Greenhouse career page URLs with a `board` query parameter (e.g. `?board=weights_and_biases&gh_jid=...`), use the `board` value as the primary Greenhouse API slug candidate. This handles cases where the hosting domain differs from the Greenhouse board (e.g. CoreWeave hosting W&B jobs).
- For Greenhouse acknowledgment/confirmation single-checkbox fieldsets (e.g. AI use policies, data consent), auto-check them at runtime using `force=True` since the native checkbox input is visually hidden behind custom-styled SVG checkmarks. Match by checking `fieldset` / `.field-wrapper` inner text for fragments like "acknowledge", "confirm", "i agree", "i accept".
- For Greenhouse single-select demographic options (Gender Identity, Race, etc.), click the label element to trigger Greenhouse's JS single-select handler rather than force-checking the input directly, which bypasses the click event and leaves form state inconsistent.
- For Greenhouse GDPR demographic data consent checkboxes, use `*=` (contains) selectors for `gdpr_demographic_data_consent` since forms may wrap the input name as `job_application[data_compliance][gdpr_demographic_data_consent_given]`.
- For Greenhouse acknowledgment fields that contain "current employer" in their label text but whose only option is "I acknowledge" or similar, guard against false-positive current-company matching with `_is_acknowledgment_field`.
- For Greenhouse work authorization questions, handle both `multi_value_single_select` and `multi_value_multi_select` field types. Add work authorization, sponsorship, and acknowledge patterns to `separately_handled_labels` so runtime-filled fields are not flagged as unknown.
- For Greenhouse standalone "country" questions, fill from `application_profile.md` but exclude questions containing "authorization", "authorisation", "authorized", "authorised", "sponsorship", "sponsor", "visa", or "countries" to avoid conflating country and work-auth questions.
- For Greenhouse comboboxes already filled with a valid (non-placeholder) value by the main payload steps, skip re-filling during the runtime demographic pass to avoid overwriting a correct LLM-generated answer.
- For Greenhouse deterministic question handling, treat "preferred first name" and "preferred last name" questions as deterministic fields answered from `master_resume.md`, and treat standalone "ethnicity" labels (not just "race ethnic" compound labels) as demographic questions handled from `application_profile.md`.
- For Greenhouse field interactions, handle native selects, modern React/ARIA comboboxes, autocomplete controls, and plain free-form text fields rather than assuming one fixed page/component type.
- For direct `job-boards.greenhouse.io/...` pages, keep the direct URL instead of rewriting it to an embed endpoint, and click the visible top-level `Apply` CTA when the inline form has not mounted yet. Treat runtime-filled profile controls such as `Location (City)` as known once answered so they do not reappear as unknown-question blockers.
- For Greenhouse select2-backed native `<select>` controls, set the underlying selected option exactly rather than fuzzy-matching placeholder choices such as `--`, and keep the visible select2 label in sync so required yes/no questions do not post as blank.
- For Greenhouse location selectors, handle state-only option sets even when the label sounds like a city question. If the profile location is `San Francisco, CA` and the live choices are states, resolve the answer to `California` instead of typing the city and leaving the combobox unselected.
- For Greenhouse async select2/autocomplete searches such as school selectors, wait for the filtered option list to populate and click the exact live result row instead of accepting the first substring match that appears.
- Do not synthesize a standalone Greenhouse `country` payload step unless the live form actually renders a country control. Fill runtime-discovered country/location controls only when they are visible on the page, and once a runtime-discovered location or country selection is verified, record it as a confirmed filled field rather than leaving it in `planned_but_unconfirmed_fields`.
- For Greenhouse explicit state-list residency gates such as `Do you live in one of the following states?`, answer from the candidate's actual current state in `application_profile.md`, not from the generic `Live In Job Location` default.
- For Greenhouse salary/compensation comfort gates such as `Are you comfortable interviewing for the salary outlined in the job description?`, answer from the explicit salary-comfort default in `application_profile.md`.
- For Greenhouse explicit free-text salary/compensation expectation prompts and explicit undergraduate/Bachelor's GPA prompts, skip LLM generation and fill the field directly from the shared `Compensation Expectations` and `Undergraduate GPA` values in `application_profile.md`. If either shared value is missing, stop through the existing pending-user-input / planned-unconfirmed path instead of leaving the field blank.
- For Greenhouse classic embed forms, handle required `question_option_id` selects such as work-authorization, commute-proximity, and in-office-availability gates, including city-specific `Yes, San Francisco` style options. Do not treat raw bracketed demographic field names like `job_application[demographic_answers]...` as text-entry steps; let the visible demographic-question flow answer those controls instead.
- For Greenhouse classic education sections, fill one visible education row deterministically from `master_resume.md` using the candidate's real school, degree, discipline, and end year. Target the live `_0` controls and active select2 widgets inside the rendered application form rather than hidden template rows or disabled scaffolding nodes.
- For Greenhouse education selects, prefer the Wharton MBA entry with `Finance` discipline when the form supports it. If the degree dropdown does not offer an MBA option, fall back to a master's-level option with `Computer Science` discipline. Do not accept unsafe partial school matches such as `Bloomsburg University of Pennsylvania` when the intended school is `University of Pennsylvania`; fail closed instead of silently picking the wrong school.
- If a Greenhouse job post exposes an `education_config` and every education field is marked `hidden`, do not synthesize school/degree/discipline/year payload steps from the resume and do not block submit on those hidden education controls.
- For Greenhouse visible demographic-question groups such as Gender Identity, Race/Ethnicity, Pronouns, Sexual Orientation, Veteran Status, Disability Status, racial/ethnic background, transgender identity, or veteran/active-member prompts, map answers from `application_profile.md` and the board's explicit decline option when no configured value exists. If `application_profile.md` already has a `Transgender Status` value, use it directly instead of declining. Target the real nested `.demographic_question` groups before falling back to broader container scans so classic multi-question EEOC sections do not collapse into one missed block.
- For Greenhouse modern self-ID sections that render visible React comboboxes inside generic `.select` containers, including multi-select-style widgets for Gender Identity, racial/ethnic background, or Sexual Orientation, fill and confirm the visible widgets themselves rather than relying on hidden legacy EEOC backing fields such as `#gender` or `#race`. If the visible self-ID section stays blank, treat it as unconfirmed and fail closed before submit.
- For Greenhouse cover-letter questions, prefer attaching the existing cover-letter file when the board exposes both file-upload and manual-entry controls. On classic `attach-or-paste` uploaders, use the attach path even when the underlying file input is only materialized after clicking the button. If the run uses a manual-entry textarea, reveal the `or enter manually` control and confirm the field is actually filled before allowing submit. If a cover-letter slot exists, do not silently skip it just because the board marks it optional.
- For Greenhouse resume and cover-letter uploads on the newer `file-upload` widget, wait for the page to finish hydrating before interacting, prefer the visible `Attach` chooser flow over writing directly to the raw hidden file input, and only treat the upload as confirmed once the widget visibly shows the chosen filename. A bare `input.files` match is not enough for those widgets.
- For Greenhouse runtime-only optional self-ID widgets, treat them as non-blocking. Autofill them only when they were parsed into explicit payload steps; do not let fallback discovery of optional voluntary self-ID controls block or stall the submit flow.

### Ashby
- For Ashby field interactions, handle visible Yes/No button groups, radio/value-select groups such as metro-area questions, preferred-office or `Office location of choice` prompts, onsite-office-attendance prompts like `Are you willing and able to come into our downtown SF office 3 days per week?`, nested phone text-message consent radios, standard EEOC value-select questions, insurance-background yes/no gates, and long textarea answers directly instead of typing into hidden checkbox inputs or timing out on slow sequential keystrokes. Map office-location selects from the configured city/location in `application_profile.md` before treating them as unknown required questions. If Ashby shows both a visible reCAPTCHA and validation errors, treat the validation errors as the first-class failure to fix before entering a manual CAPTCHA wait. Treat passive reCAPTCHA badges as non-blocking and only enter a manual wait when an active challenge is visible. Count success states such as `Your application was successfully submitted` as website confirmations.
- For Ashby explicit free-text salary/compensation expectation prompts and explicit undergraduate/Bachelor's GPA prompts, route `policy.text_value` into `String` / `LongText` steps before any generated-answer fallback. If the shared compensation or GPA source is missing from `application_profile.md`, write `submit/pending_user_input.json` and stop instead of silently leaving the field unresolved.
- For Ashby Boolean fields rendered as Yes/No button groups with a hidden `<input type="checkbox">`, clicking the visible button may only add a CSS `_active` class without actually checking the hidden checkbox. After clicking a Boolean button choice, verify the hidden checkbox state matches the intended value and force-set it via JavaScript with `change`/`input` event dispatch if needed.
- For Ashby forms with duplicate-label fields (e.g. two "Are you currently authorized..." fields with different UUIDs), disambiguate by the field's unique path/UUID using `_field_entry(page, label, path=path)` rather than relying on label text alone.
- For Ashby invisible reCAPTCHA rejection ("We couldn't submit your application"), classify as `captcha_required` for graceful skip rather than treating it as a validation error.
- For Ashby Number field types, extract digits from LLM-generated text answers. For Ashby Boolean field types, map LLM answers ("Yes"/"True"/"1") to the Yes/No button interaction.
- For Ashby "I confirm" consent patterns, add to the privacy/consent label matching alongside "I understand and agree", "acknowledge", etc.
- For Ashby and other React-controlled forms, LongText/textarea field values may be cleared when subsequent choice/button fills trigger React re-renders. The shared pipeline runs a Phase 2b verification pass after all steps are filled that re-checks and re-fills any required LongText or long (>=400 char) textarea fields that were emptied by re-renders. Board-specific `_fill_text_value` functions should also verify the value was retained after filling and fall back to JavaScript native setter + event dispatch if `fill()` alone is silently dropped by React.
- Canonicalize direct Ashby `jobs.ashbyhq.com/.../application` URLs back to the posting URL before scraping so JD extraction reads the full posting rather than the thinner application shell.
- If an Ashby or same-site company careers page lands on a thinner application shell without the JD, search same-site canonical or linked pages for the full posting before giving up on extraction.

### Workday
- For Workday (`*.myworkdayjobs.com`) applications, the autofill uses a custom multi-page browser pipeline (not the shared single-form `run_browser_pipeline`) because Workday requires account authentication and uses a sequential wizard. The canonical auth recovery order is sign in first, then password reset via the `gws` CLI, then create account as the final fallback, with email verification via `gws` if the create-account flow requires it. Explicit credential rejection is the only Workday outcome that should land as durable `auth_failed`; maintenance/service-interruption pages should become `service_unavailable` and auto-retry, while exhausted gateway states such as plain `Create Account / Sign In` or reset pages should stop as `auth_unknown` with rich evidence (`auth_state`, `auth_scope`, last attempted step, heading, alert text, visible CTAs, saved screenshot/page text). Pages are detected by `data-automation-id` attributes and filled sequentially: My Information â†’ My Experience (resume upload) â†’ Work Experience/Education (accept auto-parse) â†’ Voluntary Self-ID â†’ Custom Questions (LLM-generated) â†’ Review â†’ Submit.
- Workday `My Information` can render `How Did You Hear About Us?` and the follow-up `Source` field as either native `<select>` controls or searchable prompt widgets. For LinkedIn-sourced jobs, try the exact `LinkedIn` leaf first, then fall back to truthful social-network variants such as `Social Networking`, and verify the selected pill/aria text actually commits before continuing. Dispatch real `input`/`change` events after native-select fills so the page clears its required-field validation.
- Workday reruns can land directly on an in-progress `/apply` page or on a job page with `Continue Application`. Treat both as valid application re-entry states instead of bouncing the run into a generic auth artifact.
- Workday draft reruns must fail closed at Review. Treat explicit review roots or visible submit controls as a review shell even if heading detection drifts, capture the pre-submit screenshot, and never let `_click_next_button()` activate `Submit` or `Submit Application`.
- If the Workday page loop stays stuck for three iterations with visible validation errors, write `submit/application_submission_result.json` with a page-specific failure type such as `my_information_validation` or `my_experience_validation` instead of letting the queue collapse the run into a generic retries-exhausted stop.
- If a Workday create-account/sign-in/password-reset step raises while the page still classifies as an auth gate, convert that into the existing Workday auth artifact/result flow so the run stops as `auth_unknown` (or the more specific auth state) with evidence instead of surfacing only a generic browser failure.
- Canonicalize Workday `*.myworkdayjobs.com` and `*.myworkdaysite.com` URLs by stripping tracking query params (e.g. `?source=LinkedIn`) before scraping. For `myworkdayjobs.com`, the company slug is in the subdomain (e.g. `factset` from `factset.wd108.myworkdayjobs.com`). For `myworkdaysite.com`, the company slug is in the path (e.g. `snapchat` from `wd1.myworkdaysite.com/en-US/recruiting/snapchat/snap/job/...`). Prefer URL-derived company slugs over JD-text-derived slugs, since Workday page titles like "FactSet Careers" produce incorrect slugs.
- For Workday URLs where scrapling/stealth fetching returns empty SPA shells, fall back to the Workday CXS JSON API (`/wday/cxs/{company}/{site}/job/...`) for JD extraction. This works for both `myworkdayjobs.com` and `myworkdaysite.com` patterns.

### Lever
- Canonicalize Lever `jobs.lever.co/.../apply` URLs back to the posting URL (strip `/apply` suffix and query params like `?source=LinkedIn`) before scraping so JD extraction reads the full posting rather than the thinner application form. The autofill script reconstructs the `/apply` URL from the canonical posting URL for form filling.

### Dover
- For Dover apply URLs on `app.dover.com`, use the public `application-portal-job` API for JD extraction instead of the Cloudflare-gated HTML shell. For Dover submits, mirror the real frontend payload: send multipart form data to `application-portal-inbound-application`, keep LinkedIn / resume / phone fields at the top level, include only `CUSTOM` questions inside `application_questions` JSON, and forward `rs` / `referrerSource` from the job URL as `referrer_source`.

### Cross-Board Rules
- **Google session check:** `ensure_google_session()` runs before every autofill. Checks `ListAccounts` endpoint. If expired in headed mode, navigates to Google sign-in, sends macOS notification, and waits up to 5 min. In headless mode, logs warning and continues. Improves reCAPTCHA v3 scores.
- **Browser minimization:** Headed browsers start minimized. On macOS, new Playwright Chromium windows default to AeroSpace workspace `Q` (the user's `alt-q` space; override with `JOB_ASSETS_AEROSPACE_WORKSPACE`) so background automation stays out of the active space. The browser stays backgrounded unless manual action is required. When captcha/manual challenges appear, the runtime sets the title to `[Captcha] {Company} â€” {Role}`, sends the macOS notification, and raises that window once; the web UI "Focus Browser" button can raise it again later if needed.
- **Shared browser profile:** All workers share a single Playwright persistent profile (`~/.job-assets/playwright-submit-profile`). No per-worker subdirectories. Set up via `uv run python scripts/setup_browser_profile.py` (Google sign-in). Re-run if session expires.
- For shared pipeline submit confirmation polling, when a `validation_error` state is detected after clicking submit, wait 2 seconds and re-check the page before breaking out of the polling loop. Forms may briefly flash a validation error before transitioning to the success page (transient validation). Always take a fresh page snapshot in the final check phase rather than relying on the cached snapshot from the polling loop.
- **Captcha wait (headed browsers)**: Submit/approve runs default to headed browsers. If captcha is detected, the browser stays open for manual solving (up to `JOB_ASSETS_CAPTCHA_TIMEOUT` seconds, default 3600). macOS notification sent and the `[Captcha]` window is unminimized/raised once so the user can act immediately. Job status transitions to `awaiting_captcha`. Signal file `submit/awaiting_captcha.json` bridges subprocess â†” orchestrator. Auth artifacts (Workday, iCIMS) are not captchas. For Workday specifically, only explicit credential rejection is a true auth failure; maintenance pages auto-retry and ambiguous gateway states stop with saved evidence instead of masquerading as captcha or bad-password failures.
- After every successful application submission, reply to the confirmation email from the company (reply to self) with the pre-submit screenshot image and the autofill report markdown attached, but send that reply at most once per confirmed submit attempt. Automatic reruns and worker post-submit reconciliation must reuse the same submit-attempt reply state instead of sending a second self-reply for the same submission.
- If a real confirmation is only discovered later from `submit/application_confirmation_email.json` during disk reconciliation, backfill the same self-reply once from the preferred confirmed submit bucket when `confirmation_email_reply.json` is still missing.
- After every successful application submission, if the job was sourced from LinkedIn (the `source_url` or pipeline meta contains a `linkedin.com` URL), mark it as applied on LinkedIn by calling `mark_linkedin_job_applied(linkedin_url)` from `scripts/url_resolver.py`. This navigates to the LinkedIn posting and clicks through the "Did you apply?" dialog. If no LinkedIn source URL is available, skip silently. Log success/failure but do not fail the overall submission on a LinkedIn mark error.
- For any submitter, write a generic `submit/application_submission_result.json` artifact immediately after a website-confirmed submit. Then look for the matching confirmation email via `gws`, and sync the role into the Notion tracker once both website and email confirmations exist. If the email or Notion auth is missing, write status artifacts under the active submit-attempt bucket (`submit/` or a confirmed `submit-*` reapply directory) and surface the exact rerun command instead of silently skipping the sync.
- When the user explicitly asks to reapply for a role, create one fresh `submit-*` attempt directory for that reapply and make it the active submit artifact bucket before rerunning browser automation. Reuse that same directory across retries until the submission is completed, then clear the active pointer so a later explicit reapply gets its own new `submit-*` directory. Do not overwrite the previous completed submit attempt's artifacts, and reuse matching cached submit artifacts such as generated answers or Greenhouse application HTML from earlier attempts when the live inputs have not changed.
- After a submit click, poll for the matching confirmation email in parallel with browser-state checks. Carry the submit-start timestamp into the shared Notion-sync flow and only accept Gmail confirmations that are both new enough and plausibly tied to the same company or board, with explicit company or role evidence rather than generic provider-only phrasing, so unrelated or neighboring application emails cannot satisfy a new submit run. If the browser confirmation lags but the email confirmation arrives, treat the application as confirmed and continue the shared Notion-sync flow instead of timing out on the page shell.
- If `submit/application_submission_result.json` already records a website-confirmed application but `submit/notion_sync_status.json` is still pending, rerunning `job-assets submit --submit`, `job-assets pipeline --submit`, or `job-assets notion-sync` must skip the duplicate live resubmit and resume only the shared email/Notion reconciliation. Standalone sync reruns should derive their minimum acceptable Gmail timestamp from the recorded website confirmation or submission artifact and clear stale `submit/application_confirmation_email.json` artifacts when no fresh match exists.
- For generated application-form answers, allow optional conditional follow-up fields (for example `If other...` or `If yes...`) to remain blank without failing validation, and do not turn blank optional answers into autofill steps for hidden controls. When a required conditional follow-up question (e.g. "If yes to the above question, what role...") gets an empty LLM-generated answer because the parent answer makes it inapplicable, auto-fill "N/A" instead of failing validation. This applies to patterns like `If yes...`, `mentioned above`, `please confirm...above`, across all boards.
- When `submit_application.py` encounters an unsupported job board URL, log a JSON artifact to `submit/unsupported_board.json` and return exit code 0 so batch runs continue. Materials are still generated for manual application.
- When `submit/application_answers.json` already exists and the live question list is unchanged, reuse the cached generated answers instead of invoking the provider again during submit retries.
- For submit-time generated answers that reference directly linked public resources, fetch those resources in repo-controlled code before provider invocation and keep submit-mode provider browsing disabled. Save the current-attempt proof under `submit/linked_resource_context.json`, `submit/linked_resource_failures.json`, and `submit/linked_resource_evidence/`, and mirror the provenance into `submit/application_answers.json`. Required linked-resource fetch failures must fail closed into `pending_user_input`; optional linked-resource fetch failures may leave the answer blank but must stay visible in review artifacts.
- Across all supported board submitters, never collapse long free-text prompts into yes/no defaults just because they mention words like `experience`, `onsite`, or `location`. If the prompt is the specific onsite proceed/start/location pattern, answer it deterministically as `Yes`, the Monday in two weeks using the long date format like `March 23, 2026`, and the configured city from `application_profile.md` (for example `San Francisco`). If a long free-text prompt asks for specialized domain detail that cannot be safely inferred, write `submit/pending_user_input.json` and stop before submission instead of guessing or submitting a placeholder. Likewise, if any planned autofill field remains unconfirmed on the live form, write those exact unresolved fields into `submit/pending_user_input.json` and fail closed; do not submit until every planned field is confirmed. When this happens, also mirror the state in `submit/application_submission_result.json` as `pending_user_input` and treat the application as skipped so batch-style runs can continue to the next job while the user fills the logged answers later.
- Across all supported board submitters, route discrete positive-fit screening prompts through the shared positive-fit policy rather than board-local booleans. This includes office-attendance, commute-distance, relocation, travel, product-usage, minimum-years, and general experience/background confirmations. Degree/license/certification claims stay on the shared credential-backed exception path and must fail closed when the repo does not support the credential.
- When adding support for a new job board or company-hosted wrapper, apply the same standards already required on existing boards from the start: use `application_profile.md` defaults for deterministic yes/no and self-ID prompts, attach the prepared resume and cover letter when the board exposes those slots, confirm every planned field on the live form before submit, write pending-user-input artifacts instead of guessing on unsupported narrative questions, capture website/email/Notion confirmation artifacts, and treat those safeguards as baseline parity rather than optional board-specific hardening. New boards should import shared utilities from `scripts/autofill_common.py` (label matching, option selection, screenshot capture, page snapshotting, report writing, submit button clicking, captcha helpers) and opt into `scripts/autofill_pipeline.py` for browser-based orchestration (`autofill_main` for CLI + payload, `run_browser_pipeline` for the standard navigateâ†’fillâ†’submitâ†’confirm flow). API-based boards like Dover can use `autofill_main` with a custom `run_browser_fn` callback for submission. Board-specific logic (CSS selectors, form parsing, field inference, form filling) stays in each board's script.
- When a yes/no application prompt mentions both work authorization and sponsorship or visas in the same sentence, treat it as a sponsorship question first. Do not let the unconditional-work-authorization default flip a future-sponsorship question to `Yes`.
- For supported board submitters, answer explicit salary/compensation-comfort yes/no gates from `application_profile.md`. If a board exposes a cover-letter textarea or upload field, confirm the content/file is actually present before allowing submit; do not silently skip cover-letter slots just because the board marks them optional.
- For supported board submitters, answer explicit transgender-status yes/no prompts from the `Transgender Status` default in `application_profile.md`. The current default is `No`, and those prompts should never fall back to a generic decline answer when the profile already specifies the value.
- When `application_profile.md` specifies pronouns, submit them using equivalent board variants such as `He/Him`, `He/Him/His`, or similar label text rather than requiring an exact punctuation match.
- For explicit current-company/current-employer fields on any supported board, answer with only the current employer (`Moody's Analytics`). Do not let helper text or LLM generation expand those fields into a full company-history list.
- For long free-text browser fills, stop simulating per-character typing once the answer is large; paste/fill the value directly to avoid Playwright timeouts on cover-letter-style responses.
- Before any fresh autofill rerun on any supported board, clear the current attempt's report, review screenshots, submit debug artifacts, `application_submission_result.json`, and page screenshot directory contents from the active `submit/` directory. Keep payloads, generated answers, and unknown-question artifacts that still describe the current job, but never let stale proof from a previous attempt survive into the new run.
- For screenshot artifacts, expand nested scroll containers before capture, target the true live scroll root when stitching is necessary, and avoid duplicated top sticky headers so review screenshots show the full filled application once.
- For LibreOffice PDF builds on macOS, force the `svp` VCL backend during headless conversion and override any inherited `SAL_USE_VCLPLUGIN` value such as `osx`; relying on the shell environment can still crash `soffice` through AppKit even under `--headless`.
- For headed Playwright browser submits on macOS, do not fall back to the bundled `Google Chrome for Testing` app. Prefer installed local Chrome / Chromium / Edge or `JOB_ASSETS_BROWSER_PROVIDER=steel`, because the bundled testing browser can abort during AppKit/HIServices startup before the first page appears. If a persistent-profile local Chrome launch still aborts early on macOS, retry once without persistence and do not immediately retry the same browser app again through both the channel and executable-path fallbacks.

## LinkedIn Easy Apply

- **Board name**: `linkedin`
- **Detection**: `linkedin.com/jobs/view/` URLs where Easy Apply button is present (no external apply link)
- **Browser profile**: `.playwright-linkedin/` (NOT `~/.job-assets/playwright-submit-profile`). Requires LinkedIn login via `LINKEDIN_EMAIL`/`LINKEDIN_PASSWORD` env vars
- **Concurrency**: Serialized via `.playwright-linkedin.lock` file lock â€” one Easy Apply at a time across all workers
- **Employer naming**: LinkedIn-hosted roles must derive `company` / `company_proper` from the posting employer in the JD title or content, not from the `linkedin.com` host. Generated resume and cover-letter filenames should therefore use the employer name (`Asurion`), not `LinkedIn`.
- **Wizard**: Multi-step modal. Steps discovered dynamically (not hardcoded). Loop until "Review" or "Submit application" detected.
- **Resume step contract**: If LinkedIn shows a visible resume upload/change path, intercept the file chooser, upload the current employer-named resume, and require the live UI to expose it as the selected `Deselect resume <filename>` entry before treating the attachment as verified. If Review is reachable without any visible upload/change path, continue but report `review_without_visible_resume_controls` instead of claiming a verified fresh upload.
- **Failure artifact contract**: If LinkedIn fails before Review, write `submit/application_submission_result.json` with `status: "failed"` and a LinkedIn-specific `failure_type` such as `linkedin_modal_missing`, `linkedin_validation_loop`, `linkedin_navigation_missing`, `linkedin_timeout_after_partial_fill`, or `linkedin_resume_upload_verification_failed`. Use `submit/linkedin_submit_debug.png` as the canonical failure screenshot, and do not leave behind a fresh report or pre-submit screenshot for a review state the current attempt never reached.
- **Non-Easy-Apply truthfulness**: If LinkedIn no longer exposes Easy Apply, write `submit/application_submission_result.json` with `status: "not_easy_apply"` and `failure_type: "external_apply"` or `failure_type: "no_apply_button"`, plus the current job-page screenshot. Do not reuse `linkedin_modal_missing` when the page truthfully shows an external Apply path or no apply control at all.
- **Draft mode**: Headed browser. Fills all wizard steps, captures screenshot at review page, exits.
- **Submit mode**: Headed browser. Fills all steps, clicks "Submit application", waits for confirmation.
- **Hardcoded behaviors**: "Follow company" = always uncheck. Priority = never mark. "How did you hear" = "LinkedIn". Salary = "Open and flexible".
- **Already applied**: Detected on job page before clicking Easy Apply. Writes `application_submission_result.json` with `status: "already_applied"`.
- **Post-submit**: `mark_linkedin_job_applied()` is skipped (redundant â€” already on LinkedIn).
- **`ensure_google_session()`**: Skipped â€” separate browser profile without Google sign-in.
- Repo entrypoints auto-load `.env` and `.env.local` from the repo root. Persist local credentials such as `NOTION_API_TOKEN` in `.env.local`, keep that file untracked, and do not commit secrets.
- **"Current location" field across boards:** This field can be String (plain text), Location (autocomplete combobox), or ValueSelect (dropdown) depending on the JD. Detect `field_type` and dispatch to the correct fill kind. For boards with static step lists (SmartRecruiters, Workable), include multiple label variants ("Location", "Current location", "City") as separate optional steps â€” unmatched labels are silently skipped.
- **Ashby entry-scoped input fallback:** Ashby React forms may not set `name`/`id` attributes on custom text fields (e.g., "Current location"). When `_fillable_text_locator` fails to find `input[name="{path}"]`, fall back to finding any visible `input:not([type=hidden])` or `textarea` inside the field entry container (already located by label). This prevents silent skips where the report says "filled" but the form field is empty.
- **Greenhouse fuzzy match shortest-first:** When `_match_option_label` or `matching_option_text` fall back to substring matching, collect all matches and prefer the shortest. Prevents "United States Minor Outlying Islands" from winning over "United States".
- **Greenhouse HTML title placeholder:** Greenhouse's static HTML has `<title>page_title</title>` as a JS placeholder. When saving the raw HTML before JS executes, skip this literal string in `_candidate_page_titles()` so the Notion sync uses the proper "JD Title | Company" format instead.
- **Post-submit confirmation screenshot:** `{board}_autofill_post_submit.png` is captured at every confirmation exit point in `autofill_pipeline.py` (email confirmation, page state confirmation, captcha-then-confirm, validation-retry-then-confirm). Shown in a dedicated "Confirmation" tab in the web UI (separate from the pre-submit "Screenshot" tab). Only visible for jobs with `submitted` status.
