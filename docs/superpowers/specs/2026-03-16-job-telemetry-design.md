# Job Telemetry & Metrics System

**Date:** 2026-03-16
**Status:** Draft

## Summary

Add full telemetry and metrics to the job application pipeline. Track phase-level timing, field-level correction rates, manual interventions, and auto-fix attempts. Surface these metrics in a new TUI Stats screen and CLI report command.

## Motivation

The current system logs events at phase transitions and has some timing data (asset generation duration, submit duration), but lacks:
- Consistent phase-level timing across all 6 pipeline phases
- Field-level tracking of corrections (which fields needed manual help)
- Pre-computed per-job metrics (error rate, intervention count, total duration)
- Aggregate views (jobs/time, failure rate, board-level error rates)
- A dedicated stats/reporting UI

## New Database Tables

### `job_phase_durations`

Tracks wall-clock time for every phase a job passes through. One row per phase attempt (a job that retries submit gets multiple rows).

```sql
CREATE TABLE IF NOT EXISTS job_phase_durations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    job_id INTEGER NOT NULL REFERENCES jobs(id),
    phase TEXT NOT NULL,  -- resolve, generate, submit, fix, retry, post_submit
    started_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    ended_at TIMESTAMP,
    duration_ms INTEGER,
    exit_code INTEGER,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX IF NOT EXISTS idx_phase_durations_job_id ON job_phase_durations(job_id);
CREATE INDEX IF NOT EXISTS idx_phase_durations_phase ON job_phase_durations(phase);
```

### `field_corrections`

One row per field that was corrected by the user. Captures what was wrong and where the fix happened.

```sql
CREATE TABLE IF NOT EXISTS field_corrections (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    job_id INTEGER NOT NULL REFERENCES jobs(id),
    field_name TEXT NOT NULL,
    original_value TEXT,
    corrected_value TEXT,
    correction_source TEXT NOT NULL,  -- draft_review, submission, content_edit
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX IF NOT EXISTS idx_field_corrections_job_id ON field_corrections(job_id);
CREATE INDEX IF NOT EXISTS idx_field_corrections_source ON field_corrections(correction_source);
```

### `job_metrics`

Pre-computed rollup per job. Updated incrementally as events happen.

```sql
CREATE TABLE IF NOT EXISTS job_metrics (
    job_id INTEGER PRIMARY KEY REFERENCES jobs(id),
    total_fields INTEGER DEFAULT 0,
    fields_corrected INTEGER DEFAULT 0,
    field_error_rate REAL DEFAULT 0.0,  -- fields_corrected / total_fields
    manual_interventions INTEGER DEFAULT 0,  -- count of user actions (approve, reject, regenerate, content edits)
    auto_fix_attempts INTEGER DEFAULT 0,
    total_duration_ms INTEGER DEFAULT 0,  -- created_at to completed_at
    phase_count INTEGER DEFAULT 0,
    retry_count INTEGER DEFAULT 0,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## Instrumentation Points

### 1. Pipeline Orchestrator (automatic phase timing)

Wrap each of the 6 phases in `pipeline_orchestrator.py` with start/end timing:
- **Phase 1: resolve** — URL detection and aggregator resolution
- **Phase 2: generate** — Asset generation with provider fallback chain
- **Phase 3: submit** — Form autofill (draft or live)
- **Phase 4: fix** — Auto-fix via claude CLI
- **Phase 5: retry** — Retry with Playwright recording
- **Phase 6: post_submit** — Notion sync + email reply

Each phase writes a row to `job_phase_durations` with `started_at`, `ended_at`, `duration_ms`, and `exit_code`. Use a context manager or helper function to avoid boilerplate:

```python
def _track_phase(conn, job_id, phase):
    """Context manager that writes a job_phase_durations row."""
    phase_id = _start_phase(conn, job_id, phase)
    try:
        yield
    finally:
        _end_phase(conn, phase_id, exit_code=...)
```

Also update `job_metrics.phase_count` and `job_metrics.total_duration_ms` after each phase.

### 2. Draft Review (TUI corrections)

When the user approves a draft in the TUI:
- Load the original autofill payload (the JSON/form data generated by the autofill script)
- Load the current state (after user edits)
- Diff field-by-field. Any changed field → insert into `field_corrections` with `correction_source='draft_review'`
- Update `job_metrics`: increment `fields_corrected`, recalculate `field_error_rate`
- Increment `job_metrics.manual_interventions`

When the user rejects: log event, increment `manual_interventions`.
When the user regenerates: log event, increment `manual_interventions`.

### 3. Submission Autofill (field success/failure tracking)

After autofill completes in `submit_application.py`:
- The autofill scripts already generate `autofill_report.md` (or `.txt`) with field-level results
- Parse this report to extract: total fields attempted, fields that failed/were skipped
- Write failed/skipped fields to `field_corrections` with `correction_source='submission'`
- Update `job_metrics.total_fields` with the total field count

### 4. Content File Editing (resume/cover letter corrections)

When content files are first generated by `run_pipeline.py`:
- Snapshot `resume_content.json` and `cover_letter_text.txt` (store hash or full copy in output dir as `.original` files)

When submission starts:
- Compare current file content against the `.original` snapshot
- For `resume_content.json`: JSON diff to identify changed fields (bullet text, summary, etc.)
- For `cover_letter_text.txt`: simple hash comparison (changed = 1 correction, unchanged = 0)
- Write changed fields to `field_corrections` with `correction_source='content_edit'`
- Increment `job_metrics.manual_interventions`

## job_metrics Rollup

`job_metrics` is updated whenever:
- A phase completes → accumulate `total_duration_ms`, increment `phase_count`
- A retry happens → increment `retry_count`
- An auto-fix runs → increment `auto_fix_attempts`
- A field correction is logged → increment `fields_corrected`, recalculate `field_error_rate`
- A manual action happens in TUI → increment `manual_interventions`
- Total fields are discovered during submit → set `total_fields`

Use an `ensure_job_metrics(conn, job_id)` helper that INSERT OR IGNOREs the row before any update.

## TUI Stats Screen

New screen accessible via `s` keybinding from any screen.

### Summary Panel (top)
- Jobs processed in last 1h / 24h / 7d / all time
- Success rate (submitted / total)
- Failure rate (failed / total)
- Manual intervention rate (jobs with manual_interventions > 0 / total)
- Average field error rate across all jobs
- Average total duration per job

### Per-Job Metrics Table (middle)
Sortable DataTable with columns:
- Job ID, Company, Role, Status, Total Duration, Fields Corrected/Total, Error Rate %, Fix Attempts, Manual Interventions
- Click a row → push existing JobDetailScreen

### Phase Breakdown (bottom)
- Average duration per phase across all jobs (resolve, generate, submit, fix, retry, post_submit)
- Slowest phase highlighted
- Per-board breakdown of error rates (which boards cause the most field corrections)

### Dashboard Integration
Add to existing dashboard summary bar:
- `Error Rate: X%` (average field_error_rate)
- `Avg Duration: XmYs` (average total_duration_ms)

## CLI Report Command

Extend `job-assets report`:
- `job-assets report` — summary stats (same data as TUI Stats summary panel)
- `job-assets report --board <board>` — filtered by board
- `job-assets report --job <id>` — per-job breakdown with phase durations and field corrections list
- `job-assets report --csv` — CSV export of per-job metrics table

Reuses the same query functions that power the TUI Stats screen.
